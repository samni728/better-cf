# 产品设计

## 核心用户

核心用户是使用 Shadowrocket、OpenWrt、自建节点或类似代理/网络工具的个人运维用户。用户希望在自己当前网络环境中找到最快的 Cloudflare Anycast IPv4 / IPv6，并通过固定域名调用这些优选 IP。

典型用户不是要管理复杂 CDN 平台，而是要解决一个很具体的问题：

```text
我的网络环境下，哪些 Cloudflare IP 现在最快？
我怎样把这些 IP 自动写入自己的域名？
我怎样确认 Cloudflare DNS 已经同步成功？
```

## 核心痛点

当前 CLI 工具已经能跑出优选 IP，但存在几个问题：

- 需要手动执行。
- 每次只能看终端输出，不方便追踪历史。
- 不能批量更新 Cloudflare DNS。
- 不知道写入 DNS 的 IP 是什么时候测出来的、用了多久、是否仍然稳定。
- 如果脚本直接操作 DNS，缺少“只操作指定域名”的安全边界和同步确认。

## 产品定位

本项目定位为：

```text
Cloudflare BetterIP DNS Sync
```

它不是泛用 DNS 管理器，也不是复杂代理面板。它只专注一件事：

```text
把当前网络测出来的优选 Cloudflare IP，安全、可追溯地同步到指定域名。
```

## MVP 第一条闭环

MVP 只保留一个主用户、一个主场景、一个主数据流：

- 主用户：当前 VPS / 节点拥有者。
- 主场景：每天或手动扫描优选 Cloudflare IP，并同步到指定域名。
- 主数据流：测速参数 -> 扫描结果 -> DNS 替换 -> Cloudflare 反查确认 -> WebUI 展示。
- 主交付结果：`speed.example.com` 这类域名始终解析到最新一批经过测试的 Cloudflare A / AAAA 记录。

## Now / MVP

第一版必须做：

- 首次访问 WebUI 时创建管理员账号和密码。
- 管理员登录后才能访问配置、扫描、DNS 同步和历史记录。
- WebUI 配置目标域名、Zone ID、Cloudflare API Token。
- WebUI 支持 DNS 目标策略：
  - 单域名模式：IPv4 A 和 IPv6 AAAA 都写入同一个域名。
  - 分离域名模式：IPv4 和 IPv6 可以分别写入不同域名。
  - 分离域名模式下，每个目标可以继承统一 Cloudflare 凭据，也可以单独配置 API Token、Account ID、Zone ID。
- WebUI 配置扫描参数：
  - IPv4 / IPv6
  - TLS / 非 TLS
  - 目标结果数量
  - 设置带宽 Mbps
  - RTT 并发数
- 强制新扫描 IPv4 / IPv6。
- 保存完整测速结果：
  - 设置带宽
  - 实测带宽
  - 峰值速度
  - RTT 延迟
  - 数据中心
  - 总耗时
- 只删除指定域名下的 A / AAAA 记录。
- 批量创建新的 A / AAAA 记录。
- 写入后重新查询 Cloudflare，确认 DNS 记录与目标 IP 列表一致。
- WebUI 展示本次是“强制新扫描”还是“重测当前 IP”。
- WebUI 展示每个 IP 使用多久、连续通过天数、是否已同步 Cloudflare。
- SQLite 记录运行日志、测速结果、DNS 操作日志。
- WebUI 提供“立即执行”按钮，手动触发一次当前配置的任务。
- WebUI 提供更灵活的定时设置：
  - 每小时运行。
  - 每天固定时间运行。
  - 每 N 天固定时间运行。
- Dashboard 和运行页展示最近任务、定时任务状态和可展开日志。
- 每次执行都要保存运行日志，支持展开查看后台实时输出，方便排查问题。

管理员账号是单机部署 MVP 的安全边界。第一版不做多用户和权限矩阵，只保留一个管理员。

## Next

第二阶段做：

- 定时任务：每天早上或晚上自动强制扫描。
- 当前 DNS IP 复测：只重测已写入 Cloudflare 的 IP。
- 补位扫描：复测通过数量不足时，只扫描补足缺口。
- DNS 快照：更新前保存旧记录。
- 手动回滚到上一次 DNS 快照。
- 历史运行详情页和 IP 历史页。

## Later

后续可考虑：

- 多域名、多任务配置。
- 多 Cloudflare Zone。
- 多 VPS 对比测速。

- 更复杂的评分权重和稳定性策略。

## 产品原则

- DNS 操作必须可解释、可追溯、可确认。
- 默认强制新扫描，保证每次写入的是新测试结果。
- 复测是辅助分析，不替代强制刷新。
- 删除操作必须严格限制在指定完整域名的 A / AAAA 记录上。
- 所有敏感配置只在本地保存，不上传第三方。
- 第一版不做复杂多租户、权限矩阵和插件系统。

## Dashboard 体验原则

首页不是配置清单，而是运行结果看板。用户打开首页时，优先回答：

```text
现在是否正在执行？
执行到了哪一步？
今天更新了多少 IP？
已经写入 Cloudflare 多少 IP？
目标数量是否满足？
最近一次任务是否成功？
需要排查时日志在哪里？
```

配置项只作为健康检查和入口展示，不占据主要信息层级。

信息优先级：

1. 当前状态：待配置、已就绪、正在执行、已同步、需要处理。
2. 今日数据：更新 IP 数、写入 DNS 数、任务次数、DNS 状态。
3. 当前进度：阶段、进度条、已发现/已写入数量。
4. 目标概览：IPv4 / IPv6 域名、定时策略、下一次计划。
5. 最近任务：可展开查看实时和历史日志。
